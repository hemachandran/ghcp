🤖 GitHub Copilot Agents

The core best practice for creating custom GitHub Copilot agents is to define highly focused, specialized roles using structured .agent.md configuration files located within your repository's .github/agents/ directory. Instead of treating an agent as a generic developer, you must carefully constrain its scope, declare explicit tooling, and enforce precise boundaries.

📂 Structural and Configuration Design

  • Use Standard Locations: Save configuration profiles inside .github/agents/your-agent-name.agent.md.
  • Leverage Markdown Frontmatter: Define metadata tags such as name, description, and tools explicitly at the top of your markdown files.
  • Integrate Specific Tools: Declare exact functional permissions (e.g., ["read", "search", "edit", "execute"]) so your agent can systematically interact with files or external Model Context Protocol (MCP) servers.
  • Explicitly Select Models: Add a model: property inside your configuration to match the task's complexity, opting for faster models during execution and smarter models during architecture planning.

🎯 Prompt Engineering and Persona Guardrails

  • Avoid Monolithic Personas: Do not instruct your agent to be a broad "software engineer". Create modular agents—like a testing-specialist or implementation-planner.
  • Define Three-Tier Boundaries: Provide granular instructions grouped by Always Do, Ask First, and Never Do rules to prevent destructive actions or unapproved code modifications.
  • Provide Concrete Examples: Include real code blocks or strict markdown output templates inside your agent instructions to demonstrate what a successful completion looks like.
  • Detail the Tech Stack: List precise technical parameters, such as library versions and file paths, directly inside the agent's knowledge definitions.

🛠️ Execution and Workflow Integration

  • Complement Instruction Files: Ensure agent instructions align with your global .github/copilot-instructions.md rules instead of contradicting them.
  • Enforce Verification Prompts: Instruct the agent to present a clear, step-by-step technical plan to the developer before it executes any file edits.
  • Chain Agents via Handoffs: Break down complex pipelines by calling micro-agents or sub-agents sequentially to solve isolated components of a broader feature task.
  • Utilize Reusable Skills: Package framework-specific patterns, test schemas, or organizational commands inside .github/skills/ folders following the open agentskills.io standard.

📋 Advanced Agent Profile Blueprint

Use the following structured format inside your .agent.md file to maximize agent execution quality:

markdown 
---
name: database-optimizer
description: Analyzes repository schemas, identifies performance bottlenecks, and builds isolated migrations.
tools: ["read", "search", "edit"]
model: sonnet-4.5
---

You are a database optimization specialist. 
Your core objective is to improve query latency without altering active domain entities.

### Core Responsibilities
- Review database structural definitions for missing indexes.
- Isolate query optimizations into specific migration files.

### Operational Guardrails
- **Always Do**: Generate a standalone markdown explanation summarizing the performance delta.
- **Ask First**: Propose structural schema modifications before altering any schema files.
- **Never Do**: Modify application controller code or delete existing columns.

### Tech Stack Context
- Postgres 16
- Prisma ORM
- Schema location: `./prisma/schema.prisma`
